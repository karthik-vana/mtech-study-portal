1a Explain the difference between a queue and a circular queue. Why is the circular queue more memory-efficient?
7MCO1L2
▼
Definition

Queue: A linear data structure following FIFO (First In, First Out) principle where elements are inserted at the rear and deleted from the front.

Circular Queue: An enhanced queue where the last position is connected back to the first position, forming a logical circle.

Key Differences
ParameterSimple QueueCircular Queue
StructureLinear arrayCircular / ring-like
Memory WasteFront positions are wasted after deletionReuses freed positions
Overflow Conditionrear == MAX-1 (even if front spaces are free)rear == (front-1 + MAX) % MAX
EfficiencyLow (memory loss)High (full utilization)
ImplementationSimpleUses modulo arithmetic
Reset Required?Yes, needs queue resetNo reset required
Why Circular Queue is Memory-Efficient

In a simple queue, once front pointer advances, the vacated positions can never be reused, causing "false overflow" — the rear pointer reaches the end even though there is free space at the front.

Circular queue resolves this by wrapping the rear pointer:

rear = (rear + 1) % MAX; // Wraps around front = (front + 1) % MAX; Enqueue: rear = (rear + 1) % MAX; arr[rear] = data; Dequeue: front = (front + 1) % MAX; return arr[front]; Full: (rear + 1) % MAX == front Empty: front == rear
ASCII Diagram
Simple Queue (after 2 deletions): [_][_][C][D][E] ↑front=2 ↑rear=4 — slots 0,1 WASTED Circular Queue (same scenario): ┌──────────────────────┐ │ [E][_][_][C][D] │ (all slots reusable) └──────────────────────┘ front=3 rear=0
Conclusion: Circular queue eliminates false overflow and achieves 100% utilization of allocated memory, making it significantly more memory-efficient than a linear queue.
1b How would you reverse a single linked list? Write the algorithm and explain its working.
8MCO1L1
▼
Concept

Reversing a single linked list means making each node's next pointer point to its previous node. The original head becomes the tail and the original tail becomes the new head.

Algorithm (Iterative — 3-Pointer Method)
Algorithm REVERSE_LIST(head): prev ← NULL curr ← head next ← NULL WHILE curr ≠ NULL DO: next ← curr.next // Step 1: Save next node curr.next ← prev // Step 2: Reverse the link prev ← curr // Step 3: Move prev forward curr ← next // Step 4: Move curr forward END WHILE head ← prev // New head is last node RETURN head
Step-by-Step Trace
Original: 1 → 2 → 3 → 4 → NULL Iteration 1: prev=NULL, curr=1, next=2 1.next = NULL, prev=1, curr=2 Iteration 2: prev=1, curr=2, next=3 2.next = 1, prev=2, curr=3 Iteration 3: prev=2, curr=3, next=4 3.next = 2, prev=3, curr=4 Iteration 4: prev=3, curr=4, next=NULL 4.next = 3, prev=4, curr=NULL Result: 4 → 3 → 2 → 1 → NULL head = 4
Recursive Algorithm
Algorithm REVERSE_RECURSIVE(node): IF node == NULL OR node.next == NULL: RETURN node // Base case rest ← REVERSE_RECURSIVE(node.next) node.next.next ← node // Last node points back node.next ← NULL // Break original link RETURN rest // New head
Complexity
  • Time Complexity: O(n) — visits each node once
  • Space Complexity: O(1) iterative / O(n) recursive (call stack)
Conclusion: The iterative 3-pointer method is optimal with O(n) time and O(1) space. It is preferred over recursion to avoid stack overflow for large lists.

7MCO1L2
▼
Linked List Traversal

Traversal means visiting each node starting from the head node and following the next pointer until NULL is reached.

Algorithm TRAVERSE(head): curr ← head WHILE curr ≠ NULL DO: PRINT curr.data curr ← curr.next END WHILE
Finding Middle Element — Two-Pointer (Floyd's) Method

Slow pointer moves 1 step; Fast pointer moves 2 steps. When fast reaches end, slow is at middle.

Algorithm FIND_MIDDLE(head): slow ← head fast ← head WHILE fast ≠ NULL AND fast.next ≠ NULL DO: slow ← slow.next // 1 step fast ← fast.next.next // 2 steps END WHILE RETURN slow.data // slow is at middle
Trace Example
List: 1 → 2 → 3 → 4 → 5 → NULL Step slow fast 0 1 1 1 2 3 2 3 5 3 3 NULL (stop) Middle = slow.data = 3 ✓
Complexity
  • Time: O(n) — single pass
  • Space: O(1) — no extra memory
Advantage: Single-pass O(n) algorithm avoids counting nodes first (no two-pass needed). For even-length list, returns second middle node.

3b How would you detect a loop in a circular linked list? Provide the algorithm with a clear explanation.
8MCO1L1
▼
Floyd's Cycle Detection Algorithm (Tortoise and Hare)

Uses two pointers: slow (moves 1 step) and fast (moves 2 steps). If a cycle exists, they will eventually meet inside the cycle.

Algorithm
Algorithm DETECT_LOOP(head): slow ← head fast ← head WHILE fast ≠ NULL AND fast.next ≠ NULL DO: slow ← slow.next fast ← fast.next.next IF slow == fast THEN: RETURN "Loop Detected" END IF END WHILE RETURN "No Loop" // Finding Loop Start: slow ← head WHILE slow ≠ fast DO: slow ← slow.next fast ← fast.next Loop start = slow.data
Trace Example
List: 1 → 2 → 3 → 4 → 5 → (back to 3) ↑__________________________| Step slow fast 0 1 1 1 2 3 2 3 5 3 4 4 ← MEET! Loop detected at node 4
Why It Works

In the cycle of length C, if slow enters cycle when fast is k steps ahead, fast reduces the gap by 1 every step (fast gains +1 on slow each iteration). They must meet within C steps.

Complexity
  • Time: O(n)
  • Space: O(1) — only two pointers
Conclusion: Floyd's algorithm is the most optimal cycle detection method — O(n) time, O(1) space — avoiding hash set approaches that need O(n) extra space.
Question Set 4
4a Define a stack and explain its LIFO property with a real-life analogy. What operations are typically performed on a stack?
7MCO1L1
▼
Definition

A Stack is a linear data structure that follows the LIFO (Last In, First Out) principle — the last element inserted is the first one to be removed. Elements are inserted and deleted from only one end called the TOP.

LIFO Property with Real-Life Analogy

Analogy: Stack of Plates

  • You add plates on top (PUSH)
  • You always take the top plate first (POP)
  • You cannot remove a middle plate without removing all plates above it
  • The last plate placed is the first plate used → LIFO

Other analogies: Pile of books, undo history in MS Word, browser back navigation.

Stack Operations
OperationDescriptionComplexity
PUSH(x)Insert element x at topO(1)
POP()Remove and return top elementO(1)
PEEK() / TOP()View top element without removingO(1)
isEmpty()Returns true if stack is emptyO(1)
isFull()Returns true if stack is full (array)O(1)
SIZE()Returns number of elementsO(1)
Array Implementation
int stack[MAX]; int top = -1; PUSH(x): if top == MAX-1 → OVERFLOW else stack[++top] = x; POP(): if top == -1 → UNDERFLOW else return stack[top--]; PEEK(): return stack[top];
Conclusion: Stack is a fundamental ADT where all operations execute in O(1) time. Its LIFO property makes it ideal for expression parsing, recursion management, and backtracking.
4b How would you implement a stack or queue using two queues or two stacks? Discuss with examples.
8MCO1L1
▼
Stack using Two Queues

Use Q1 (main) and Q2 (auxiliary). PUSH: enqueue all of Q1 to Q2, enqueue new element to Q1, then move all from Q2 back to Q1 → Q1 front is always the top of stack.

Q1=[], Q2=[] PUSH(1): Q1=[1] PUSH(2): Q2=[], enqueue 2 to Q2=[2], move Q1[1] → Q2=[2,1], swap: Q1=[2,1], Q2=[] PUSH(3): Q2=[], enqueue 3 to Q2=[3], move Q1[2,1] → Q2=[3,2,1], swap: Q1=[3,2,1] POP(): dequeue Q1 → returns 3 ✓ (LIFO!) Time: PUSH O(n), POP O(1)
Queue using Two Stacks

Use S1 (inbox) and S2 (outbox). ENQUEUE: push to S1. DEQUEUE: if S2 empty, pop all from S1 to S2, then pop from S2.

S1=[], S2=[] ENQUEUE(1): S1=[1] ENQUEUE(2): S1=[1,2] ENQUEUE(3): S1=[1,2,3] DEQUEUE(): S2 empty → move S1 to S2 S2=[3,2,1] → pop S2 = 1 ✓ (FIFO!) DEQUEUE(): S2=[3,2] → pop S2 = 2 ✓ ENQUEUE(4): S1=[4] (S2 still has [3]) DEQUEUE(): S2 not empty → pop S2 = 3 ✓ Amortized Time: ENQUEUE O(1), DEQUEUE O(1) amortized
Conclusion: Queue with two stacks is highly efficient (amortized O(1)) because each element is moved at most twice. This pattern is used in asynchronous I/O systems.

Polynomial Representation

Each node in the linked list represents one term of the polynomial, storing its coefficient and exponent.

Node Structure: | coeff | exp | next | Example: 3x³ + 5x² + 2x + 7 [3|3] → [5|2] → [2|1] → [7|0] → NULL
Algorithm to Add Two Polynomials
Algorithm ADD_POLYNOMIALS(P1, P2): result ← empty list p ← P1.head q ← P2.head WHILE p ≠ NULL AND q ≠ NULL DO: IF p.exp == q.exp THEN: sum_coeff ← p.coeff + q.coeff IF sum_coeff ≠ 0: INSERT(result, sum_coeff, p.exp) p ← p.next q ← q.next ELSE IF p.exp > q.exp THEN: INSERT(result, p.coeff, p.exp) p ← p.next ELSE: INSERT(result, q.coeff, q.exp) q ← q.next END WHILE // Copy remaining terms WHILE p ≠ NULL: INSERT(result, p.coeff, p.exp); p ← p.next WHILE q ≠ NULL: INSERT(result, q.coeff, q.exp); q ← q.next RETURN result
Example
P1: 4x³ + 3x² + 2x [4|3] → [3|2] → [2|1] → NULL P2: 5x² + 4x + 1 [5|2] → [4|1] → [1|0] → NULL Addition Step-by-Step: p.exp=3 > q.exp=2 → copy [4|3] p.exp=2 = q.exp=2 → 3+5=8, insert [8|2] p.exp=1 = q.exp=1 → 2+4=6, insert [6|1] p=NULL → copy remaining [1|0] Result: 4x³ + 8x² + 6x + 1 [4|3] → [8|2] → [6|1] → [1|0] → NULL
Complexity
  • Time: O(m + n) where m, n are the number of terms
  • Space: O(m + n) for result list

Binary Search — Concept

Binary Search works on a sorted array. It compares the target with the middle element and eliminates half the remaining search space at each step.

Recursive Algorithm
Algorithm BINARY_SEARCH(arr, low, high, target): // Base Case: Element not found IF low > high THEN: RETURN -1 mid ← (low + high) / 2 IF arr[mid] == target THEN: RETURN mid // Found at index mid ELSE IF target < arr[mid] THEN: RETURN BINARY_SEARCH(arr, low, mid-1, target) // Search left half ELSE: RETURN BINARY_SEARCH(arr, mid+1, high, target) // Search right half

Merge Sort — Overview

Merge Sort uses Divide and Conquer: split array into halves, sort each half recursively, then merge the sorted halves.

Algorithm MERGE_SORT(arr, l, r): IF l < r: mid ← (l + r) / 2 MERGE_SORT(arr, l, mid) MERGE_SORT(arr, mid+1, r) MERGE(arr, l, mid, r) Recurrence: T(n) = 2T(n/2) + O(n) Solution (Master Theorem): T(n) = O(n log n)
Complexity Comparison

Bubble SortO(n)O(n²)O(n²)O(1)Yes

Why Merge Sort is More Efficient
  1. Guaranteed O(n log n) in ALL cases — bubble/selection degrade to O(n²) in average/worst case
  2. Efficient for large datasets — O(n log n) grows much slower than O(n²)
  3. For n=10,000: Merge Sort ≈ 133,000 ops vs Bubble Sort ≈ 100,000,000 ops
  4. External sorting — works well for data that doesn't fit in RAM (stable merge)
  5. Parallelizable — both halves can be sorted simultaneously
Limitation: Merge Sort requires O(n) extra space. For in-place sorting, consider Quick Sort or Heap Sort (though Quick Sort has O(n²) worst case).

7MCO2L1

Cycle Detection using DFS (Directed Graph)

Use visited[] and recStack[] (recursion stack). If a neighbor is already in the recursion stack → cycle exists.

Algorithm DFS_CYCLE(graph, V): visited[1..V] ← FALSE recStack[1..V] ← FALSE FOR each vertex u in V: IF NOT visited[u]: IF DFS_UTIL(u, visited, recStack): RETURN TRUE // Cycle found RETURN FALSE Algorithm DFS_UTIL(u, visited, recStack): visited[u] ← TRUE recStack[u] ← TRUE FOR each neighbor v of u: IF NOT visited[v]: IF DFS_UTIL(v, visited, recStack): RETURN TRUE ELSE IF recStack[v]: RETURN TRUE // Back edge → CYCLE! recStack[u] ← FALSE RETURN FALSE
Cycle Detection using BFS — Kahn's Algorithm (DAG)
Algorithm KAHN_CYCLE(graph, V): inDegree[1..V] ← 0 FOR each edge (u,v): inDegree[v]++ Queue Q ← all vertices with inDegree = 0 count ← 0 WHILE Q not empty: u ← DEQUEUE(Q); count++ FOR each neighbor v of u: inDegree[v]-- IF inDegree[v] == 0: ENQUEUE(Q, v) IF count == V: RETURN "No Cycle" ELSE: RETURN "Cycle Exists"
Complexity
  • DFS method: O(V + E) time, O(V) space
  • BFS/Kahn's: O(V + E) time, O(V) space

Which is More Memory Efficient?

Both use O(h) space theoretically. However, iterative is practically more memory-efficient because:

  1. Recursive uses system call stack — each frame stores multiple variables (return address, local vars)
  2. Iterative stack stores only node pointers
  3. System stack is limited (typically 1-8 MB); iterative uses heap memory (virtually unlimited)

A B-tree of order m is a balanced multi-way search tree where each node can hold up to m-1 keys and have up to m children. All leaves are at the same level.

  1. Disk I/O Minimization: Each disk read brings an entire block (4KB-16KB). B-tree node = one disk block containing many keys. AVL/RB nodes contain only 1 key per read → many more I/O operations.
  2. Lower Height: B-tree of order 1000 has height ≤ 4 for 1 billion records (vs ~30 for BST). Fewer levels = fewer disk accesses.
  3. Large Branching Factor: B-tree with m=1000 accesses at most log₁₀₀₀(n) nodes. For n=10⁹: only 3 levels!
  4. All Leaves at Same Level: Guaranteed O(log n) worst-case for all operations.
  5. Sequential Access: B+ tree (variant) — all data in leaves with leaf-level links → efficient range queries and full scans.

Splay Tree — Concept

A Splay Tree is a self-adjusting BST where every accessed node is moved to the root through a series of rotations called "splaying." No explicit balance factor stored. Provides O(log n) amortized operations.

Real-World Applications
  1. Cache Implementations (LRU Cache): Most recently accessed node reaches the root. Frequently accessed data stays near top → faster future access. Used in web browser caches.
  2. Data Compression — LZW Algorithm: LZW needs to look up and insert recently used patterns. Splay tree moves recently used patterns to root → amortized O(1) for working set. Better than hash tables for temporal locality.
  3. Network Routing (IP Tables): Frequently accessed IP routes naturally bubble to root. Reduces lookup time for common destinations.
  4. Text Editors (Rope Data Structure): Splay trees used in rope data structure for string operations. Recently edited positions accessed faster.
  5. Virtual Memory Management: Page table lookups — recently accessed pages move to root of splay tree.
  6. Garbage Collectors: Memory allocators use splay trees to track free memory blocks. Most recently freed/allocated blocks remain near root.
  7. Game AI Path Caching: Recently computed paths stay near root for O(1) retrieval.
Key Advantage: Splay trees exploit temporal locality — if you access X recently, you'll likely access X again soon. No other balanced BST offers this property.

Working Set Theorem

Splay trees satisfy the Working Set Property: accessing the t-th most recently accessed element takes O(log t) time. If your application repeatedly accesses the same small set of "hot" keys, splay trees outperform both AVL and RB trees significantly.

LZW Algorithm Overview

LZW maintains a dictionary of previously seen substrings (patterns). During compression, it repeatedly looks up the longest matching pattern in the dictionary, outputs its code, and adds the new pattern (extended by one character).

Why LZW Favors Splay Trees
  1. Temporal Locality: In real text, the same substrings appear repeatedly ("the", "and", common words). Splay tree moves these to the root → subsequent accesses O(1) vs O(log n) for AVL/RB.
  2. Working Set Dominance: LZW dictionary lookup follows 80/20 rule — 20% of patterns account for 80% of accesses. Splay keeps this hot 20% near root.
  3. Dynamic Dictionary Growth: LZW continuously adds new patterns. Splay tree handles insertions with the same amortized O(log n) as AVL/RB, but with no color/height bookkeeping overhead.
  4. No Extra Storage: Splay tree needs no color bits or height fields → lighter memory footprint for large LZW dictionaries (up to 4096 entries in standard LZW).
  5. Access Frequency Adaptation: As compression proceeds, more common patterns naturally settle near the root — the tree self-optimizes for the input data's character distribution.
Concrete Example
Compressing: "ABABABAB..." LZW Dictionary after initial processing: "A"→65, "B"→66, "AB"→256, "BA"→257, "ABA"→258... Splay Tree after 100 compressions: "ABA"(root) ← most accessed / \ "AB" "B" / "A" ... other patterns deeper Search "ABA": O(1) — already at root! Search "AB": O(1) — left child of root! vs AVL/RB: O(log 4096) ≈ 12 comparisons every time
Conclusion: Splay trees are ideal for LZW because LZW exhibits strong temporal locality — a small subset of patterns dominates. Splay's self-optimization reduces average lookup time from O(log n) to O(1) for the working set, significantly speeding up compression/decompression.
