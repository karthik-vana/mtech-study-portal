Q1(a) Define Cloud Computing and list some of its key characteristics.
7M
▶

📌 Definition

Cloud Computing is a model for enabling anytime, anywhere, on-demand network access to a shared pool of computing resources (networks, servers, storage, applications, services) that can be quickly set up and removed with very little effort. (NIST SP 800-145)

✅ Key Characteristics (NIST)

  • On-demand Self-service: Users provision resources automatically without human interaction with the provider.
  • Broad Network Access: Capabilities available over the network via standard mechanisms (PC, mobile, tablet).
  • Resource Pooling: Provider's resources pooled to serve multiple consumers via multi-tenancy; physical/virtual resources dynamically assigned.
  • Rapid Elasticity: Capabilities can be elastically set up/created/released to scale rapidly with demand.
  • Measured Service: Resource usage monitored, controlled, and reported — pay-per-use model.
  • Scalability: Scale up/down compute resources based on workload.
  • Fault Tolerance: Redundant systems ensure high availability.

Q1(b) What are the primary technologies used for network-based systems in computing?
8M
▶

📌 Overview

Network-based computing systems rely on a set of core technologies that enable communication, resource sharing, and service delivery across distributed environments.

✅ Primary Technologies

  • Client-Server Model: Centralized server provides resources/services; clients make requests over network. Foundation of web, email, FTP.
  • Peer-to-Peer (P2P): Every node acts as both client and server. No central authority. Example: BitTorrent, blockchain.
  • Service-Oriented Architecture (SOA): Software components expose services via standard interfaces (SOAP/REST). Enables loose coupling.
  • Web Services: SOAP, WSDL, UDDI — enable interoperable machine-to-machine communication over HTTP.
  • REST APIs: Stateless, HTTP-based, JSON/XML — cornerstone of modern cloud APIs.
  • Message-Oriented Middleware (MOM): Asynchronous communication via message queues (RabbitMQ, AWS SQS).
  • Remote Procedure Call (RPC/gRPC): Execute procedures on remote systems as if local. Used in microservices.
  • Virtualization: VMware, KVM — hides the complexity of physical hardware to enable multi-tenancy.
  • TCP/IP Protocol Suite: HTTP, FTP, DNS, SMTP — foundational network protocols.
  • SDN (Software-Defined Networking): Programmable network control for dynamic resource allocation.

Q2(a) Demonstrate the concept of Distributed Computing and how it is beneficial over centralized computing.
7M
▶

📌 Definition

Distributed Computing is a computing method where multiple independent computers that talk to each other over a network to do a common task — they look like one single system to the user.

✅ Core Concepts

  • Concurrency: Many tasks run at the same time on different computers (nodes).
  • No Shared Memory: Computers (nodes) exchange information by sending messages to each other — there is no shared memory.
  • Transparency: The system hides details like where data is stored (location), how data is copied (replication), how failures are handled, and how tasks are moved (migration) — all transparent to the user.
  • Openness: Standard communication protocols allow different types of computers and software to work together without issues.
  • Scalability: Scale out by adding more computers (nodes) to the system instead of upgrading a single machine.

Q2(b) Explain the role of system models in Distributed and Cloud Computing.
8M
▶

📌 Definition

System Models are simplified pictures or descriptions that show the important features and working of distributed/cloud systems, giving a clear structure for designing, studying, and understanding these systems.

✅ Types of System Models

  • Physical Model: Describes the physical hardware — computers (nodes), connections (links), and how the network is arranged. It shows what the actual machines look like.
  • Architectural Model: Defines how different parts of the system talk to each other (client-server, P2P, layered). It focuses on the roles of each part and how they communicate.
  • Fundamental Model: Covers how messages are sent (with/without time limits), how failures are handled (crash, message loss, bad behavior), and how security is managed.
  • Interaction Model: Synchronous — messages must arrive within a fixed time limit. Asynchronous — messages can arrive at any time (used in cloud systems).
  • Failure Model: Crash failure (node stops working), Omission failure (messages lost), Byzantine failure (node behaves incorrectly). Cloud systems should handle crash and omission failures.
  • Security Model: Identifies possible attacks — listening to private data, changing data, pretending to be someone else — and the solutions to stop them (TLS encryption, IAM access control).

Q3(b) Explain how Grid Computing works and provide an example of its application.
7M
▶

📌 Definition

Grid Computing is a distributed computing method that coordinates geographically dispersed different types of resources (different hardware and software) from multiple organizations or institutions to solve large-scale computational problems.

✅ How Grid Computing Works

  • Resource Discovery: Special grid management software (like Globus Toolkit) automatically finds and lists all available resources (computers, storage) across different organizations.
  • Job Scheduling: The work/job is split into smaller tasks and sent to available computers across the grid. Schedulers like PBS and Condor decide which machine does which task based on what is available.
  • Authentication: Security certificates (PKI) are used to verify identity when accessing resources across different organizations.
  • Data Management: Grid FTP (a fast file transfer tool) is used to move large amounts of data between computers in the grid efficiently.
  • Execution: Large jobs are broken into smaller tasks. These tasks are sent to computers that are free (idle). Once done, all results are collected and combined.
  • Virtual Organization (VO — a group of people/institutions sharing resources): A Virtual Organization is a group of people or institutions from different places who collaborate and share each other's computing resources.

Q4(b) How does Utility Computing refer to in the context of modern computing?
8M
▶

📌 Definition

Utility Computing is a service allocation/setup model in which computing resources — CPU, memory, storage, and bandwidth — are offered as a metered service and billed based on how much you actually use, similar to how you pay for electricity or water.

✅ Key Characteristics in Modern Context

  • Metered Billing: Resources charged per unit (per GB-hour, per vCPU-hour). Example: AWS EC2 On-Demand pricing.
  • On-Demand Access: No long-term commitments; create or remove resources instantly.
  • OPEX Model: Converts IT spending from a large one-time investment (buying servers — called CAPEX) to a regular monthly expense (renting cloud resources — called OPEX).
  • SLA Guarantees: The cloud provider promises a specific level of service (like uptime and response time) through an SLA (Service Level Agreement). If they break this promise, they must pay a penalty.
  • Resource Abstraction: Users don't manage underlying hardware — provider handles infrastructure.
  • Modern Examples: AWS Lambda (pay per invocation), Azure Functions, Google Cloud Run — serverless is peak utility computing.

💡 Key Conclusion

In modern computing, utility computing underpins the cloud economics model. Serverless computing (e.g., AWS Lambda) is the most advanced form of utility computing — you pay only for the exact time your code runs, with no charges when it is idle.

📌 Key Components

  • Nodes/Processes: Individual computing units (servers, VMs, containers) that execute tasks. Communicate via message passing.
  • Communication Network: Physical/logical links connecting nodes (LAN, WAN, Internet). Protocols: TCP/IP, UDP, HTTP.
  • Middleware: Software layer abstracting heterogeneity — CORBA, RMI, gRPC, message brokers (Kafka).
  • Naming Service: Maps logical names to physical addresses — DNS, LDAP, ZooKeeper.
  • Synchronization: Logical clocks (Lamport clocks), distributed transactions, consensus protocols (Paxos, Raft).
  • Replication Service: Maintains data copies across nodes for fault tolerance and performance.
  • Security Service: Authentication (Kerberos), authorization (RBAC), encryption (TLS).
  • Resource Manager: Allocates and manages computational resources — schedulers (YARN, Mesos).

8M

📌 Types of Virtualization

  • Server Virtualization: A single physical server is divided into multiple VMs using a hypervisor. Each VM gets its own operating system, CPU, and memory — as if it were a separate computer. Types include: Full virtualization (VMware ESXi — complete hardware simulation), Para-virtualization (Xen — modified OS for better performance), and OS-level virtualization (Docker containers — shares the OS kernel).
  • Storage Virtualization: Multiple physical storage devices (hard drives, SSDs) are combined and presented as one large logical storage space. Users see one big storage pool instead of many separate drives. Examples: SAN (Storage Area Network), NAS (Network Attached Storage), AWS EBS, VMware VSAN.
  • Network Virtualization: Multiple physical network devices (switches, routers) are combined using software to form one or more virtual networks. This allows better control and flexibility. Technologies used: VLANs (virtual local networks), VPNs (secure tunnels), SDN (programmable networking), and VXLAN (overlay networks for cloud).
  • Desktop Virtualization (VDI): The desktop (what the user sees and uses) runs on a powerful central server, and the user accesses it remotely from any device — no need to install software on the user's machine. Example: Citrix Virtual Apps, AWS WorkSpaces.
  • Application Virtualization: Applications run in their own isolated environment (container or virtual space) without needing to be fully installed on every machine. Example: Docker containers or the Java Virtual Machine (JVM). Example: Docker containers, Java JVM.
  • Memory Virtualization: RAM (memory) from multiple servers is combined into one large shared pool, and workloads can use memory from any server in the pool as if it were local memory.

Q3(a) Explain how virtualization enhances resource utilization in cloud environments.
8M
▶

📌 Definition

Virtualization enhances resource utilization by abstracting physical hardware and enabling multiple virtual machines to share a single physical server, greatly increasing how much of the hardware is actually being used — from just 10–15% in physical servers to 60–80% with virtualization.

✅ How Virtualization Enhances Utilization

  • Server Consolidation: Instead of running one application per physical server (which wastes most of the hardware), you can run 10 or more VMs on one server — this eliminates idle hardware. For example, 10 physical servers can be replaced by 1 server running 10 VMs.
  • Dynamic Resource Allocation: The hypervisor automatically changes how much CPU and memory each VM gets based on how much it currently needs. VMware DRS (Distributed Resource Scheduler) does this automatically across multiple hosts.
  • Workload Balancing: Running VMs can be moved from a busy server to a less busy server automatically (using VMware vMotion) to balance the load — without stopping the VM or affecting users.
  • Resource Overcommitment: The hypervisor can assign more virtual CPU/memory to VMs than what physically exists, because not all VMs use their full allocation at the same time. Special drivers (memory balloon drivers) take back unused memory from idle VMs and give it to busy ones.
  • Snapshot and Cloning: A VM can be copied (cloned) instantly to create a duplicate for testing or development — no need to buy new hardware.
  • Power Management: When certain servers are not being used much, they are automatically powered off to save electricity (VMware DPM — Distributed Power Management handles this).

📌 Definition

Cloud APIs (Application Programming Interfaces) are software interfaces (access points) that let users and programs set up, configure, and manage cloud resources automatically — without clicking manually in a web console. REST APIs are the dominant standard.

✅ Roles of APIs in Cloud

  • Resource Provisioning: Users can create or delete VMs, storage, and networks by sending simple API requests (e.g., AWS EC2 RunInstances API) — no manual steps needed.

  • Monitoring & Management: CloudWatch API fetches metrics; Auto Scaling API adjusts capacity.
  • Cross-Cloud Interoperability: Standardized APIs (OpenStack, Kubernetes) enable multi-cloud portability.
  • Security: APIs use IAM tokens, OAuth 2.0, API keys for authentication.

📌 Definition

Virtualization is the process of using software to create a virtual (fake but functional) version of real hardware — such as servers, storage, networks, and operating systems — using a hypervisor software layer that hides complexity.

✅ Significance in Cloud Computing

Q1(c) What is a distributed trust algorithm in the context of cloud computing?
5M
▶

📌 Definition

A Distributed Trust Algorithm is a way of building and managing trust between different computers (nodes) in a distributed cloud system — without needing one central authority (like a single trusted server) to approve everything. It ensures all parties can verify each other's identity and interactions securely.

✅ Key Points

  • Challenge: In multi-cloud or federated setups, no single trusted authority exists for all computers. So, trust is calculated by combining the opinions of multiple trusted nodes — this is called distributed trust.
  • Web of Trust: Each computer evaluates how much to trust another based on its own past experience with that node, plus recommendations received from other trusted nodes.
  • Byzantine Fault Tolerance (BFT): BFT consensus algorithms (like PBFT — Practical Byzantine Fault Tolerance) allow the system to work correctly even if some nodes are faulty or behave maliciously. The system needs at least 2/3 of nodes to be honest for it to function correctly.
  • Blockchain-based Trust: Blockchain uses an unchangeable record (ledger) to store trust-related data. Smart contracts (self-executing code on blockchain) automatically enforce the trust rules.
  • PKI: PKI (Public Key Infrastructure) uses a chain of digital certificates to establish trust in a step-by-step hierarchy — widely used in cloud systems for HTTPS/TLS secure connections.

💡 Key Conclusion

Distributed trust algorithms are very important for cloud federation (multiple clouds working together), IoT (Internet of Things), and blockchain — systems where a single central authority is not practical or safe. PBFT and chain-of-trust (PKI) are the most commonly used approaches in cloud security.

Q3(a) Apply cloud storage best practices to ensure data redundancy and durability in a cloud environment.
7M
▶

📌 Key Best Practices

  • Erasure Coding / Data Replication: AWS S3 achieves extremely high durability (11 nines = 99.999999999%) by automatically making multiple copies of your data across at least 3 different Availability Zones (AZs), using erasure coding (a method of splitting data into pieces with error-correction information).
  • Cross-Region Replication (CRR): Cross-Region Replication (CRR) automatically copies your data to a bucket in a different geographic region. This protects your data even if an entire region (like a whole data center city) is affected by a disaster.
  • Versioning: Versioning keeps a history of every version of each file. If you accidentally delete or overwrite a file, you can easily restore any previous version. This is important for data protection.
  • Lifecycle Policies: Lifecycle policies automatically move data to cheaper storage tiers as it is accessed less frequently. For example: Frequently used data stays in S3 Standard → less-used data moves to S3-IA (Infrequent Access) → rarely-used data goes to Glacier (archive storage). This saves money without manual effort.
  • Encryption: Cloud storage should always be encrypted. Options: SSE-S3 (AWS manages the keys using AES-256 encryption), SSE-KMS (you manage your own keys using AWS Key Management Service), or SSE-C (you provide your own encryption keys for maximum control).
  • Immutable Backups: S3 Object Lock (WORM = Write Once, Read Many) prevents anyone from deleting or modifying files for a set time period. This is very important for compliance and protection against ransomware attacks.
  • Checksums: Checksums (like MD5 or SHA-256) are used to verify that data has not been corrupted during upload, download, or storage over time. If the checksum does not match, data corruption is detected immediately.

💡 Key Conclusion

Using multiple layers of protection — replication across AZs, cross-region replication, versioning, and immutable backups (Object Lock) — cloud storage can meet the strictest enterprise requirements, including zero data loss (RPO = 0). AWS S3's 11 9s durability comes from these layered redundancy mechanisms.

📌 IAM Solution Design

  • Principle of Least Privilege: Every user and role should be given only the minimum permissions needed to do their job — nothing more. Using wildcard (*) in policies (which grants all permissions) should be avoided in production environments.
  • RBAC (Role-Based Access Control): Instead of giving permissions to each user one by one, define roles for each department (e.g., DevOps Role, Finance Role, HR Role). Assign users to the appropriate role — they automatically get the permissions of that role. This is much easier to manage.

  • Service Roles: AWS services like EC2 and Lambda should use IAM Roles (temporary, automatically refreshed credentials) when they need to access other services — never store access keys directly in the application code.
  • MFA Enforcement: MFA (Multi-Factor Authentication) should be required for all sensitive actions and for logging into the cloud console. MFA means the user must verify their identity using two methods — for example, a password plus a phone app code (like Google Authenticator).
  • Permission Boundaries: Permission Boundaries set the maximum level of permissions a delegated administrator can grant to others. This prevents a situation where someone gives themselves or others more permissions than they are supposed to have (called privilege escalation).
  • CloudTrail Audit: AWS CloudTrail records every API call made in your account — who did what, when, and from where. This log is essential for security audits, compliance reporting, and investigating security incidents.

📌 VM Security in Multi-Tenant Environments

In a multi-tenant cloud, many customers' VMs share the same physical server. VM security mechanisms make sure that one customer's VM cannot access, read, or affect another customer's VM — preventing data theft and cross-tenant attacks.

✅ Key VM Security Mechanisms

  • Hypervisor Isolation: Type 1 hypervisors like ESXi and KVM are installed directly on the hardware and enforce very strict separation between VMs. One VM cannot read the memory or run processes belonging to another VM — this is the most basic form of tenant isolation.
  • Dedicated vCPUs (CPU Pinning): Side-channel attacks like Spectre and Meltdown can potentially let one VM spy on another VM running on the same physical CPU. Dedicated vCPUs (by pinning physical CPU cores to specific VMs) prevent this. AWS offers 'Dedicated Instances' (single-tenant hardware) and 'Dedicated Hosts' (full physical server control) for this purpose.
  • Virtual Network Isolation: Each customer's VMs are placed in their own private network (VPC) with their own IP address range. Security Groups act like a personal firewall for each VM — blocking all traffic except what is explicitly allowed.
  • Encrypted VM Storage: EBS (Elastic Block Store) volumes are encrypted using AES-256 encryption. All stored data is protected — even the cloud provider's own staff cannot read a customer's encrypted data without the correct encryption key.
  • VM Image Security: Use minimal, hardened VM images (AMIs) that include only the software needed for the job. Disable all unnecessary services and ports. Regularly check VMs against the CIS Benchmark (a security standard) to find and fix security gaps.
  • EDR/AV Agents: Install EDR (Endpoint Detection and Response) or antivirus agents inside VMs to detect suspicious activities — such as a compromised VM trying to attack other VMs (lateral movement) or secretly mining cryptocurrency using your cloud resources.

💡 Key Conclusion

VM security works at multiple levels: the hypervisor keeps VMs separate at the hardware level, VPCs and Security Groups provide network isolation, encryption protects stored data, and security agents inside VMs detect threats. Together, these layers create a strong, multi-layer defense (called defense-in-depth) against attacks in multi-tenant environments.

📌 Security Governance Definition

Security Governance is the collection of policies (rules), procedures (step-by-step processes), standards (benchmarks), and controls (technical safeguards) that describe how an organization manages, monitors, and improves its overall security. It includes who is responsible for security, how risks are handled, and which compliance rules must be followed.

✅ Key Principles of Cloud Security Governance

  • Shared Responsibility Model: In the Shared Responsibility Model, the cloud provider is responsible for securing the physical infrastructure (hardware, network, data center). The customer is responsible for securing their data, access controls, and applications. Security governance defines these boundaries clearly so there is no confusion about who is responsible for what.
  • Risk Management Framework: Risk Management Frameworks like NIST RMF and ISO 27005 guide organizations through a structured process: Identify risks (e.g., data breach, exposed APIs) → Assess how serious they are → Mitigate them with controls → Monitor them continuously.
  • Compliance Management: Organizations must show that their cloud setup meets specific legal and industry rules — for example, GDPR (European data privacy), HIPAA (US health data), PCI-DSS (payment card security), SOC 2 (service organization security), and ISO 27001 (information security). Tools like AWS Artifact and Azure Compliance Manager provide ready-made compliance documents and certificates as evidence.
  • Policy Enforcement: Policy enforcement tools like AWS SCPs (Service Control Policies) and Azure Policy automatically prevent users from creating resources that violate security or compliance rules — for example, blocking the creation of unencrypted S3 buckets or VMs in restricted regions.
  • Continuous Monitoring: SIEM (Security Information and Event Management) tools like Splunk and Microsoft Sentinel collect and analyze security logs from all cloud services in real time. They detect suspicious activities (potential attacks) and generate the evidence needed for compliance audits.
  • Security Auditing: Security governance requires regular reviews: quarterly access reviews (check who has which permissions), penetration testing (simulate attacks to find weaknesses), and vulnerability assessments (scan for security gaps). These cannot be done once and forgotten — governance mandates they happen on a regular schedule.
  • Incident Response Policy: Defined IR playbooks for cloud incidents (data breach, ransomware) with clear escalation paths.

💡 Key Conclusion

Good security governance changes security management from reactive (responding to incidents after they happen) to proactive (preventing incidents before they occur). Organizations that implement strong governance frameworks — with clear policies, automated enforcement, and continuous monitoring — experience fewer security breaches and can achieve compliance certifications faster.

This affects all customers sharing that physical host.

The top cloud security challenge is misconfiguration and IAM weakness — not provider-side attacks. The CSA "Cloud 9" threat model shows that organizations, not providers, are responsible for 95%+ of cloud security failures.

✅ Strategies to Reduce Cloud Security Risks

  • Zero Trust Architecture: Zero Trust means: never automatically trust any request just because it comes from inside the network. A strong cloud security posture uses multiple defense layers working together: Zero Trust (never assume trust), strong IAM (only necessary access), encryption (protect data always), CSPM (detect misconfigurations automatically), and continuous monitoring (detect threats in real time). Security should be designed into the cloud architecture from the very beginning — it is much harder and riskier to add security after the system is already built.

📌 Fault Tolerance in Cloud

Fault Tolerance is the ability of a cloud system to continue working normally even when one or more of its parts (servers, storage, network links) fail. This is achieved by building in redundancy (extra copies of resources), replication (copying data to multiple locations), and automatic failover (switching to backup systems instantly when a failure is detected).

✅ Key Fault Tolerance Mechanisms

  • Data Replication: Data can be replicated in two ways: Synchronous replication (like RDS Multi-AZ — data is written to the primary and standby at the same time before confirming success). This guarantees zero data loss (RPO = 0) but adds a small delay (latency). Asynchronous replication (like S3 CRR — data is written to the primary first, then copied to the backup shortly after). This has near-zero data loss risk and no performance impact on the primary.
  • Load Balancing: Load Balancers (ALB — Application Load Balancer, ELB — Elastic Load Balancer) continuously check the health of each instance. If an instance fails its health check, the load balancer immediately stops sending traffic to it — within seconds. The remaining healthy instances handle all the traffic, so users don't notice any disruption.
  • Auto Scaling: AWS Auto Scaling constantly monitors the health and load of your instances. If an instance fails or if traffic increases beyond a threshold, new instances are automatically launched to replace or supplement them — without any manual action. The system heals itself.
  • Circuit Breaker Pattern: In a microservices setup, one service may depend on many other services. If a downstream service (one being called) fails, the Circuit Breaker Pattern detects this and 'opens the circuit' — stopping new requests from reaching the failed service. This prevents the failure from spreading to other parts of the system (cascading failure), keeping the overall system stable.
  • Chaos Engineering: Chaos Engineering is the practice of deliberately causing failures in a production system (like Netflix's Chaos Monkey, which randomly terminates cloud instances) to test whether the system can survive and recover. The goal is to find and fix weaknesses before real failures occur.

📌 EC2 Internet Security Implications

✅ External Threats

  • Port Scanning & Brute Force: As soon as an EC2 instance gets a public IP address, it is immediately detected by internet scanners (like Shodan). Automated bots start trying to break in by guessing passwords on common ports — SSH (port 22 for Linux) and RDP (port 3389 for Windows) — within minutes of the instance being launched.
  • Exploit Attacks: If your EC2 instance is running outdated or unpatched software (like an old version of Apache web server or OpenSSH), attackers can exploit known vulnerabilities to run malicious code on your instance (called Remote Code Execution — RCE). Even patched systems face risk from zero-day vulnerabilities (newly discovered bugs with no fix yet).
  • DDoS: Internet-facing EC2 instances can be targeted by DDoS attacks that flood them with so much fake traffic that they become unable to serve legitimate users. AWS Shield (Standard — free, Advanced — paid) automatically detects and blocks many DDoS attacks at the network level.
  • MITM Attacks: If your EC2 instance sends or receives data without encryption, attackers in the network path (Man-in-the-Middle attack) can intercept and read or modify that data. All services must use TLS (HTTPS) to encrypt communications.

✅ Internal Threats

  • Lateral Movement: If one EC2 instance is compromised (taken over by an attacker), the attacker can use it as a base to attack other instances inside the same VPC (Virtual Private Cloud). This is called lateral movement. Unrestricted security group rules (allowing all traffic between instances) make this easier for attackers.
  • Metadata Service Exploitation: SSRF (Server-Side Request Forgery) is an attack where a hacker tricks the EC2 instance into making a request to the internal metadata URL (169.254.169.254), which can expose the IAM role credentials (access keys) attached to the instance. Mitigation: Use IMDSv2 (Instance Metadata Service v2), which requires a session token and prevents SSRF-based credential theft.
  • Crypto-mining: A compromised EC2 instance can be secretly used to mine cryptocurrency (crypto-mining) — consuming your cloud compute resources and running up your bill, while the attacker benefits from the mined coins.

✅ Mitigation Strategies

  • Set Security Group rules to allow inbound traffic only from known, trusted IP addresses and specific ports. Never use 0.0.0.0/0 (open to the entire internet) for SSH or RDP ports.
  • Instead of opening port 22 for SSH, use AWS Systems Manager Session Manager — it provides secure shell access through the AWS console without opening any inbound ports on the EC2 instance.
  • Enforce IMDSv2 for metadata access, enable AWS GuardDuty (threat detection service that monitors for suspicious activity), and keep AWS CloudTrail enabled (records all API actions for auditing).

💡 Key Conclusion

Any EC2 instance with a public IP address faces constant attack attempts from the internet. Strong security must be configured from the moment the instance is launched — not added later. Use: minimal security group rules, IMDSv2 for metadata safety, Systems Manager instead of open SSH, AWS WAF for application protection, and GuardDuty for threat monitoring.

📌 Security Rules in EC2

EC2 security is controlled at two layers: Security Groups (instance-level, stateful firewall) and Network ACLs (subnet-level, stateless). Both operate on transport and application layer protocols.

✅ Transport Layer (Layer 4) Security Rules

  • TCP Rules: Allow specific TCP ports (443 for HTTPS, 22 for SSH, 3306 for MySQL). Deny all others. Restricts which services are accessible.
  • UDP Rules: Allow UDP traffic only for essential services: port 53 for DNS (translates domain names to IP addresses) and port 123 for NTP (keeps server clocks synchronized). Restrict these to known, trusted source IP addresses only.
  • ICMP Rules: Allow ICMP (ping) traffic only from specific management or monitoring IP addresses. This prevents ICMP-based attacks like ping floods and Smurf attacks, which can overload network devices.
  • Stateful SGs: Security Groups are stateful — when you allow outbound traffic, the response traffic is automatically allowed back in. This simplifies firewall rule management and prevents issues where return packets are accidentally blocked.

✅ Application Layer (Layer 7) Security Rules

  • WAF Rules: HTTP/HTTPS inspection — block SQL injection (';DROP TABLE), XSS (<script>), CSRF, rate limiting at application layer.
  • ALB Rules: ALB (Application Load Balancer) can route web requests to different target groups based on the URL path, HTTP headers, or hostname. This allows you to separate traffic between different microservices and apply different security rules to each one.
  • TLS Termination: The ALB handles TLS termination — it decrypts the incoming HTTPS traffic so that backend servers do not need to manage encryption individually. It enforces minimum TLS 1.2 (older, weaker versions are blocked), controls which encryption algorithms are allowed (cipher suites), and manages SSL/TLS certificates automatically using AWS Certificate Manager (ACM).

✅ Impact on Overall Security

  • Transport layer (Layer 4) rules restrict which ports are open, significantly reducing the number of ways an attacker can try to enter the system.
  • Application layer (Layer 7) WAF rules protect against the OWASP Top 10 most critical web application security risks — such as injection attacks, broken authentication, and security misconfigurations.
  • Using both transport layer rules (Security Groups) and application layer rules (WAF) together creates defense-in-depth — multiple layers of protection that an attacker would have to break through one by one.

💡 Key Conclusion

Security rules at both the transport layer (Security Groups/NACLs) and the application layer (WAF) create a multi-layered defense for EC2 workloads. Transport rules control who can connect (which IPs and ports). Application rules block attacks on the allowed connections. Both layers together are essential for full cloud application security.

📌 AAA Framework

The AAA (Authentication, Authorization, Accounting) framework is the security model used by cloud platforms to control and audit access to resources.

✅ Authentication

  • Who are you? Authentication is the process of confirming that a user or system is who they claim to be — before giving them any access.
  • Methods: Username/password + MFA (TOTP, hardware keys), SSO via SAML 2.0/OIDC (shared identity (one login used across multiple systems)), Certificate-based (mTLS for service-to-service), API Keys / IAM Roles.
  • AWS Implementation: IAM users, Cognito for app users, AWS SSO for workforce, STS for temporary credentials.

✅ Authorization

  • What can you do? Authorization decides what an already-authenticated user is allowed to do in the cloud — which services they can use, which operations they can perform, and which data they can access.
  • Common authorization models: RBAC — Role-Based Access Control (permissions based on the user's role, e.g., Admin, Developer, Viewer), ABAC — Attribute-Based Access Control (permissions based on attributes like department, location, or resource tags), MAC — Mandatory Access Control (the system enforces permissions based on security labels — used in highly regulated environments).
  • AWS Implementation: AWS implements authorization through: IAM Policies (JSON documents defining allowed/denied actions), SCPs (Service Control Policies — organization-wide rules that limit what any account can do), Permission Boundaries (maximum permission limits for delegated admins), and Resource Policies (attached directly to resources like S3 buckets to control who can access them).
  • By default, everything is denied. A user or service can only do something if there is an explicit Allow statement in their IAM policy. This 'deny by default' approach enforces the principle of least privilege.

✅ Accounting

  • What did you do? Logging all actions for audit, forensics, and billing.
  • AWS Implementation: AWS provides accounting through: CloudTrail (logs every API call made in the account — who did what and when), CloudWatch Logs (stores application and system logs), AWS Config (tracks changes to resource configurations over time), and Cost Explorer (shows detailed cloud usage and spending data).
  • Required for HIPAA/PCI-DSS/SOC 2 compliance evidence.

📌 AWS S3 Security Features

✅ Encryption Features

  • SSE-S3 (Server-Side Encryption): AWS automatically manages the encryption keys using AES-256 (a very strong encryption standard). This is the default and simplest option — you do not need to do anything to enable or manage encryption keys.
  • SSE-KMS: Instead of AWS managing the keys, the keys are stored in AWS KMS (Key Management Service). You control when keys are rotated (changed for security), and AWS CloudTrail records every time a key is used. You can also create your own customer-managed keys (CMK) for additional control.
  • SSE-C: You provide your own encryption key with every upload or download request. AWS uses the key to encrypt/decrypt the data but never stores the key on its servers. You have maximum control — but you must safely manage the keys yourself.
  • CSE (Client-Side Encryption): You encrypt the data on your own computer before uploading it to S3. AWS only ever stores the already-encrypted data and never sees the original plaintext content. This provides the highest level of security because even a breach of AWS systems would not expose your data.
  • In-Transit Encryption: HTTPS/TLS 1.2+ mandatory — bucket policy can enforce "aws:SecureTransport: true".

✅ Access Control Features

  • Bucket Policies: Bucket Policies are JSON rules attached directly to an S3 bucket. They can allow or deny access based on: IAM identity (specific users or roles), VPC endpoints (only internal cloud traffic allowed), or IP address ranges (restrict access to specific networks or offices).
  • IAM Policies: IAM Policies are attached to IAM users or roles and define exactly which S3 operations they are allowed to perform — for example, allowing a specific role to GetObject (download) from a bucket but not PutObject (upload) or DeleteObject.
  • Block Public Access: The S3 Block Public Access setting is a switch that, when turned on, prevents any S3 bucket or object from being made publicly accessible — even if an individual bucket policy or ACL tries to make it public. This is a critical safeguard against accidentally exposing sensitive data to the internet.
  • VPC Endpoint Policy: A VPC Endpoint allows resources inside your private cloud network (VPC) to connect to S3 directly without going through the public internet. You can configure bucket policies to deny all access except from your VPC endpoint — ensuring S3 is never accessed from the public internet.
  • Pre-signed URLs: A Pre-signed URL is a time-limited URL that you generate and share with someone. It grants temporary access to a specific S3 object for a limited time (e.g., 1 hour) — without requiring the recipient to have an AWS account or credentials. Very useful for securely sharing files.
  • Object Lock (WORM): S3 Object Lock (WORM — Write Once, Read Many) makes objects immutable for a set period — they cannot be deleted or overwritten by anyone, including the bucket owner. This is required for compliance with regulations like SEC Rule 17a-4 (financial records) and FINRA (brokerage records).

💡 Key Conclusion

AWS S3 offers a complete, multi-layered security system: automatic encryption by default, fine-grained access control at multiple levels (IAM, bucket policy, ACL), network isolation through VPC endpoints, and data immutability through Object Lock. Combined, these features make S3 suitable for storing even the most sensitive and regulated data — including healthcare records (HIPAA), payment data (PCI-DSS), and government information.

Cloud Standards are officially agreed rules, technical specifications, APIs (programming interfaces), data formats, and protocols that allow cloud systems from different providers to communicate and work together without compatibility problems.

✅ Why Standards Matter

  • Vendor Lock-in Prevention: When cloud providers follow common standards, you can move your application from one provider to another (e.g., AWS to Azure) using the same APIs and tools — without needing to rewrite your entire application.
  • Interoperability: Standard interfaces allow cloud systems from different providers to exchange data and services with each other. This makes cloud federation (providers working together) and hybrid cloud deployments (mixing private and public clouds) much easier to build and maintain.
  • Innovation Acceleration: When a platform becomes a standard (like Kubernetes for containers), a large ecosystem of tools grows around it. A monitoring tool, deployment pipeline, or security scanner built for Kubernetes works on AWS, Azure, and GCP — giving users huge flexibility and choice.
  • Cost Reduction: Using standard APIs reduces the cost and effort of connecting different cloud services. Standard data formats (like Parquet or Avro) eliminate the need to convert data between proprietary formats when moving between providers.
  • Compliance: International standards organizations like ISO and NIST publish security and compliance frameworks. Cloud providers implement these frameworks and get certified — making it easier for customers to meet regulatory requirements by choosing certified providers.
  • Security Baseline: Cloud-specific security standards like CSA CCM (Cloud Controls Matrix) and ISO 27017 (cloud security guidelines) define a minimum baseline of security controls that all compliant cloud providers must have — giving customers confidence that basic protections are in place.

💡 Key Conclusion

Cloud standards create a fair, open, and competitive cloud market. Kubernetes (for containers) and the S3 API (for object storage) have become de facto standards — meaning they are universally supported even without being official standards. As more providers adopt standards like OCCI, TOSCA, and CDMI, it becomes even easier to build multi-cloud applications and avoid vendor lock-in.
